# Code Review 指引

以下是 codex 执行 code review 时应遵循的准则。构建 review prompt 时，将这些要求嵌入指令中。

## 审查标准

只标记满足以下全部条件的问题：

1. 实质性影响代码的正确性、性能、安全性或可维护性
2. 问题是具体且可操作的（不是泛泛的代码风格吐槽）
3. 修复要求的严格程度与代码库整体水平一致
4. 问题是本次改动引入的（已有 bug 不标记）
5. 原作者知道后大概率会修复
6. 不依赖对代码库或作者意图的未说明假设
7. 如果认为改动会破坏其他部分，需要能指出具体受影响的代码
8. 显然不是作者有意为之的变更

## 评论规范

- 清楚说明为什么这是个 bug
- 准确传达严重程度，不夸大
- 简洁，正文最多一段
- 代码片段不超过 3 行，用 markdown 代码标记
- 明确说明触发条件（什么场景/输入下会出问题）
- 语气客观，不讨好不指责
- 让原作者一眼就能理解

## 优先级标签

在标题开头标注优先级：
- `[P0]` — 必须立刻修复，阻塞发布
- `[P1]` — 紧急，下个迭代需解决
- `[P2]` — 普通，后续修复
- `[P3]` — 低优，有则更好

## 使用 `codex review` 审查

通过 `scripts/codex_review.py` 包装脚本调用，自动获取 diff 并过滤冗余输出，只返回审查结论。自定义 prompt 只需写审查要求：

```bash
# 使用默认审查标准
python <skill-dir>/scripts/codex_review.py --uncommitted

# 传入自定义审查要求
python <skill-dir>/scripts/codex_review.py --uncommitted "只报告本次改动引入的实际 bug，忽略琐碎风格问题，每个问题标注优先级 [P0]-[P3]"

# 对比分支
python <skill-dir>/scripts/codex_review.py --base main "关注安全性和向后兼容性"
```
